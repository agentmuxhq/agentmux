## What Was Done This Session

### 1. Flat Portable Layout → Clean Layout with Launcher

Built and tested a clean portable directory structure:

```
agentmux-cef-0.32.110-x64-portable/
  agentmux.exe       325 KB   ← launcher (user double-clicks this)
  README.txt
  runtime/                    ← all internals
    agentmux-cef.exe           real CEF host
    agentmuxsrv-rs.x64.exe    backend sidecar
    libcef.dll                 251 MB CEF runtime
    frontend/                  built SolidJS app
    locales/en-US.pak
    (CEF DLLs, paks, GPU files)
```

**Why a launcher?** `libcef.dll` is a load-time dependency — the OS loader needs it before `main()` runs. `SetDllDirectoryW` in `main()` is too late. The launcher (agentmux-launcher crate, 325 KB) calls `SetDllDirectoryW("runtime/")` then spawns `runtime/agentmux-cef.exe`.

**ZIP on Desktop:** `agentmux-cef-0.32.110-x64-portable.zip` (148 MB)

### 2. File Stripping Optimizations

Updated `cef:bundle` and `cef:package:portable` Taskfile tasks:
- Stripped SwiftShader (`vk_swiftshader.dll`, `vulkan-1.dll`) — deprecated upstream, saves ~6 MB
- Stripped WebGPU DLLs (`dxil.dll`, `dxcompiler.dll`) — not used, saves ~6.5 MB
- Stripped 49 non-English locale paks — saves ~18 MB
- **Kept** GPU files: `libEGL.dll`, `libGLESv2.dll`, `d3dcompiler_47.dll`
- Result: 311 MB on disk (down from 365 MB stock), 148 MB ZIP

### 3. CEF Size Reduction Research

Wrote `specs/cef-size-reduction.md` with cross-platform analysis.

**Key finding:** The official CEF builds from Spotify CDN already use `is_official_build=true` with full optimization and ship without PDB files. The 251 MB `libcef.dll` is **already stripped** — `symbol_level=0` does NOT give 50% reduction as initially claimed. Custom builds save only ~20-40 MB from feature disables.

### 4. Benchmarks: Tauri vs CEF

Ran automated benchmarks (`scripts/bench-full.sh`, `scripts/bench-cdp.mjs`).

| Metric | Tauri (WebView2) | CEF (bundled) | Winner |
|--------|-----------------|---------------|--------|
| Disk size | 30 MB | 311 MB | Tauri |
| Startup (to sidecar) | 548 ms | 152 ms | **CEF (3.6x)** |
| Baseline RSS (1 term) | 412 MB | 352 MB | **CEF (-60 MB)** |
| Input latency (P50) | 33.3 ms | 33.1 ms | Tie |
| Scroll FPS (50K lines) | 60.0 | 60.0 | Tie |

CEF wins on startup and memory. Rendering is identical (same Chromium engine). Tauri wins on distribution size.

## What Works (CEF Build)

- CEF host starts, spawns backend sidecar with Job Object cleanup
- IPC bridge (HTTP POST + CustomEvent dispatch)
- Terminal panes, sysinfo panes, context menus (JS overlay)
- Window transparency/opacity (DWM + WS_EX_LAYERED)
- Frameless window with custom title bar drag
- Min/max/close buttons
- Pane DnD (split, move) and tab DnD (reorder)
- DevTools (remote debugging on port 9222)
- Portable packaging with file stripping
- Clean portable layout (launcher + runtime/ subdirectory)

## What Doesn't Work Yet (Phase 4)

1. **Window resize** — frameless window has no hit-test regions for edge/corner drag resize. This is the most user-visible gap. CEF Views has `is_frameless()` but needs `get_nonclient_metrics()` or manual hit-testing via `WM_NCHITTEST`.

2. **Multi-window** — `open_window_at_position`, `list_windows`, `focus_window` commands exist but aren't wired to the frontend. Need to handle `window.open()` or a custom "new window" command.

3. **Cross-window drag** — 10 drag commands are stubbed (`start_cross_drag`, `accept_cross_drag`, etc.) but not implemented. Needs CEF's `CefDragHandler` or a custom IPC-based drag protocol between windows.

4. **File drop** — dropping files into terminal panes needs `CefDragHandler::OnDragEnter` to extract file paths and forward them to the terminal as text.

5. **React cleanup** — `react`, `react-dom`, `@types/react` still in `package.json`. Some code uses `React.KeyboardEvent` and `React.RefObject` types. Needs migration to native DOM types.
