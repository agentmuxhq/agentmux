# Spec: AgentMux Isolated Claude/Anthropic Authentication

**Status:** Research / Design
**Date:** 2026-03-21
**Author:** Research via Claude Code agent

---

## Executive Summary

AgentMux can achieve fully isolated Claude authentication, completely separate from the user's personal `~/.claude/` credentials. The recommended approach is **`CLAUDE_CONFIG_DIR` isolation combined with `ANTHROPIC_API_KEY` as the primary auth path**, with OAuth via `CLAUDE_CONFIG_DIR` as a secondary option. A first-party Anthropic OAuth client ID for AgentMux is not obtainable through self-service — Anthropic explicitly prohibits third-party apps from offering claude.ai login via their own OAuth app registrations.

---

## 1. Claude Code OAuth Flow — Technical Deep Dive

### 1.1 Source Evidence

Claude Code ships as a compiled Bun binary (on Windows) and a minified Node.js bundle (`@anthropic-ai/claude-code` npm package, currently v2.1.81). The `cli.js` source (12 MB, 15,728 lines, minified) was examined directly.

### 1.2 OAuth Client ID

The production OAuth client ID is hardcoded in the CLI:

```
CLIENT_ID: "9d1c250a-e61b-44d9-88ed-5944d1962f5e"
```

This is a **shared, public client ID** registered with Anthropic's OAuth server for all Claude Code CLI installations. It is not a secret and cannot be rotated by end users. There is no separate client registration per-installation.

A staging/local development client ID also exists (`22422756-60c9-4084-8eb7-27705fd5cf9a`), but this is Anthropic-internal only.

### 1.3 OAuth Endpoints (Production)

```
CLAUDE_AI_AUTHORIZE_URL: https://claude.ai/oauth/authorize
CONSOLE_AUTHORIZE_URL:   https://platform.claude.com/oauth/authorize
TOKEN_URL:               https://platform.claude.com/v1/oauth/token
API_KEY_URL:             https://api.anthropic.com/api/oauth/claude_cli/create_api_key
ROLES_URL:               https://api.anthropic.com/api/oauth/claude_cli/roles
MANUAL_REDIRECT_URL:     https://platform.claude.com/oauth/code/callback
```

### 1.4 OAuth Scopes

The CLI requests these scopes depending on auth type:

- Console auth scopes: `["org:create_api_key", "user:profile"]`
- Claude.ai subscription scopes: `["user:profile", "user:inference", "user:sessions:claude_code", "user:mcp_servers", "user:file_upload"]`

### 1.5 Token Storage

On **Windows and Linux**, OAuth tokens are stored in plaintext at:

```
{CLAUDE_CONFIG_DIR}/.credentials.json
```

Default value of `CLAUDE_CONFIG_DIR` is `~/.claude` (built as `os.homedir() + "/.claude"`).

On **macOS**, tokens are stored in the encrypted macOS Keychain. The keychain service name is generated by function `RR()`:

```javascript
function RR(suffix = "") {
  const configDir = d1();  // resolves CLAUDE_CONFIG_DIR
  const dirHash = !process.env.CLAUDE_CONFIG_DIR
    ? ""
    : `-${sha256(configDir).substring(0, 8)}`;
  return `Claude Code${OAUTH_FILE_SUFFIX}${suffix}${dirHash}`;
}
```

**Critical finding:** When `CLAUDE_CONFIG_DIR` is set, the keychain service name gets a unique 8-character hash suffix derived from the config dir path. This means **setting `CLAUDE_CONFIG_DIR` creates a completely isolated keychain entry on macOS**. On Windows/Linux the isolation is already implicit via the file path.

### 1.6 The `.credentials.json` Structure

The file stores:

```json
{
  "claudeAiOauth": {
    "accessToken": "...",
    "refreshToken": "...",
    "expiresAt": 1234567890000,
    "scopes": ["user:inference", "user:profile", ...],
    "subscriptionType": "pro" | "max" | null,
    "rateLimitTier": "..." | null
  }
}
```

### 1.7 Environment Variables That Override the Auth Source

Authentication precedence (highest to lowest) from the source:

1. `CLAUDE_CODE_USE_BEDROCK` / `CLAUDE_CODE_USE_VERTEX` / `CLAUDE_CODE_USE_FOUNDRY` — cloud provider creds
2. `ANTHROPIC_AUTH_TOKEN` — raw bearer token sent as `Authorization: Bearer <value>`
3. `ANTHROPIC_API_KEY` — API key sent as `X-Api-Key` header (always used in `-p` non-interactive mode)
4. `apiKeyHelper` setting — shell script that outputs a key or token dynamically
5. OAuth credentials from `{CLAUDE_CONFIG_DIR}/.credentials.json` (from `claude auth login`)

Notable: `CLAUDE_CODE_OAUTH_TOKEN` env var also exists — injects a pre-fetched OAuth token directly without needing a credentials file at all.

### 1.8 The `CLAUDE_CODE_OAUTH_CLIENT_ID` Override

```javascript
let K = process.env.CLAUDE_CODE_OAUTH_CLIENT_ID;
if (K) A = { ...A, CLIENT_ID: K };
```

**This env var allows overriding the OAuth client ID used in the auth flow.** However, a custom client ID is only useful if you have a registered OAuth app at Anthropic's authorization server — which is not publicly available (see Section 2).

## 2. Third-Party OAuth App Registration — Is It Possible?

### 2.1 Official Documentation Position

The Claude Agent SDK docs (at `platform.claude.com/docs/en/agent-sdk/overview`) contain this explicit note:

> **"Unless previously approved, Anthropic does not allow third party developers to offer claude.ai login or rate limits for their products, including agents built on the Claude Agent SDK. Please use the API key authentication methods described in this document instead."**

### 2.2 No Public OAuth App Registration Portal

Anthropic does not provide a self-service OAuth app registration portal (equivalent to GitHub OAuth Apps, Slack App Directory, etc.). The `platform.claude.com/docs/en/build-with-claude/oauth` page returns 404. There is no documented process for registering a custom `client_id`.

### 2.3 What `CLAUDE_CODE_OAUTH_CLIENT_ID` Actually Enables

The env var allows redirecting the OAuth flow to use a different client ID, **but only if Anthropic has registered that client ID on their server**. Since the approval process is not public, this is effectively limited to Anthropic partners with special arrangements (e.g., the federated deployment hosts in the approved list: `claude.fedstart.com`, `claude-staging.fedstart.com`).

### 2.4 What Enterprise/Federated Deployments Get

The code references "fedstart.com" deployments — these appear to be government or regulated-industry deployments where Anthropic has registered custom OAuth endpoints. The `CLAUDE_CODE_CUSTOM_OAUTH_URL` env var enables pointing the CLI to one of these approved alternate OAuth servers, but the approved list is hardcoded and very short:

```javascript
j2K = [
  "https://beacon.claude-ai.staging.ant.dev",
  "https://claude.fedstart.com",
  "https://claude-staging.fedstart.com"
]
```

AgentMux cannot be added to this list without Anthropic cooperation.

### 2.5 Conclusion

**AgentMux cannot register its own OAuth client with Anthropic through any self-service mechanism.** The only viable self-service authentication mechanisms are:

1. **`ANTHROPIC_API_KEY`** — user provides their API key from platform.claude.com
2. **`CLAUDE_CONFIG_DIR` isolation** — user runs `claude auth login` in an AgentMux-isolated config directory, still using Anthropic's OAuth with the Claude Code client ID
3. **`apiKeyHelper`** — a script that dynamically provides a key/token

## 3. `CLAUDE_CONFIG_DIR` Isolation — Full Analysis

### 3.1 How It Works

When `CLAUDE_CONFIG_DIR` is set before running any `claude` command:

- All settings read from `$CLAUDE_CONFIG_DIR/settings.json` (not `~/.claude/settings.json`)
- Credentials stored in `$CLAUDE_CONFIG_DIR/.credentials.json` (Windows/Linux) or a unique keychain entry (macOS)
- Session history, cache, stats, plugins — all go to the isolated directory
- The `~/.claude/` directory is entirely ignored

### 3.2 For AgentMux

AgentMux should set `CLAUDE_CONFIG_DIR` to:

```
~/.agentmux/claude-auth/
```

or per-profile:

```
~/.agentmux/profiles/<profile-id>/claude-auth/
```

All `claude` child processes spawned by AgentMux must have this env var set.

### 3.3 The Auth Login Flow Under CLAUDE_CONFIG_DIR

When `claude auth login` is run with `CLAUDE_CONFIG_DIR` set:

1. CLI opens the browser to `https://claude.ai/oauth/authorize?client_id=9d1c250a-...&scope=...&redirect_uri=...&code_challenge=...`
2. User logs in with their claude.ai / Console credentials (same account they use for the web)
3. OAuth code returned, CLI exchanges for tokens at `https://platform.claude.com/v1/oauth/token`
4. Tokens stored in `$CLAUDE_CONFIG_DIR/.credentials.json`

The user is logging in to **their own Anthropic/claude.ai account** — this is indistinguishable from normal `claude auth login` from the user's perspective. The difference is the credentials end up in AgentMux's isolated directory, not in `~/.claude/`.

### 3.4 Spawning `claude -p` with Isolated Credentials

```rust
let mut cmd = Command::new("claude");
cmd.env("CLAUDE_CONFIG_DIR", agentmux_claude_config_dir)
   .args(["-p", "--verbose", "--output-format", "stream-json", "--include-partial-messages"])
   .arg(prompt);
```

The spawned `claude -p` process will:
- Read credentials from `~/.agentmux/claude-auth/.credentials.json`
- Never touch `~/.claude/`
- Operate as if it is a standalone Claude Code installation

### 3.5 Checking Auth Status

```bash
CLAUDE_CONFIG_DIR=~/.agentmux/claude-auth/ claude auth status --json
```

Returns:
```json
{
  "loggedIn": true,
  "authMethod": "claude_ai",
  "email": "user@example.com",
  "subscriptionType": "pro"
}
```

## 4. API Key Alternative — Full Analysis

### 4.1 How It Works

Setting `ANTHROPIC_API_KEY` in the environment completely bypasses OAuth. In non-interactive mode (`claude -p`), the key is **always used when present** — no approval prompt.

```rust
let mut cmd = Command::new("claude");
cmd.env("ANTHROPIC_API_KEY", api_key)
   .env("CLAUDE_CONFIG_DIR", agentmux_claude_config_dir)  // for settings isolation
   .args(["-p", "--verbose", "--output-format", "stream-json"]);
```

### 4.2 API Key Sources

Users can obtain API keys from:
- `https://platform.claude.com/settings/keys` (Anthropic Console, pay-per-use)
- Team/Enterprise Console admins can provision keys for team members

**Limitation:** API keys are associated with the **Anthropic Console** (pay-per-use) account, not with Claude.ai subscription accounts. A user with a Claude Pro/Max subscription cannot use `ANTHROPIC_API_KEY` to bill to their subscription — they'd need a separate Console account.

### 4.3 `apiKeyHelper` Script

For more dynamic credentials (e.g., tokens that rotate, or tokens fetched from a vault), `settings.json` supports:

```json
{
  "apiKeyHelper": "/path/to/script.sh"
}
```

The script outputs the key/token to stdout. Claude Code calls it every 5 minutes or on 401.

AgentMux could write this setting to `$CLAUDE_CONFIG_DIR/settings.json` and use a helper script to retrieve the key from AgentMux's own secure storage (Windows Credential Manager, macOS Keychain, etc.).

### 4.4 `CLAUDE_CODE_OAUTH_TOKEN` (Advanced)

For users with OAuth credentials, AgentMux can read the token from its isolated credentials file and inject it directly:

```rust
// Read from ~/.agentmux/claude-auth/.credentials.json
let token = read_agentmux_oauth_token()?;
cmd.env("CLAUDE_CODE_OAUTH_TOKEN", token);
// No CLAUDE_CONFIG_DIR needed if using this approach
```

However, this requires AgentMux to handle token refresh itself when the token expires. Not recommended unless the config dir approach has problems.

## 5. Existing Examples — Other Apps That Do This

### 5.1 The Claude Agent SDK

`@anthropic-ai/claude-agent-sdk` / `claude-agent-sdk` Python package — the official way to embed Claude Code capabilities in third-party apps. It uses `ANTHROPIC_API_KEY` exclusively (by design, per Anthropic policy). It does not support claude.ai OAuth for third-party apps.

### 5.2 Claude Code VS Code Extension

Shares the same `~/.claude/` config directory as the CLI. Not isolated.

### 5.3 Claude Code Desktop App

Uses its own credentials via the OS browser/OAuth flow, separate from CLI. This is a first-party Anthropic product so it has its own registered client ID.

### 5.4 Bedrock/Vertex/Azure Deployments

Enterprises route Claude through AWS/GCP/Azure. No OAuth to claude.ai — uses cloud provider credentials instead. AgentMux could support this path too by setting `CLAUDE_CODE_USE_BEDROCK=1` etc.

## 6. Recommended Architecture for AgentMux

### 6.1 Recommended Approach: Hybrid (API Key Primary, OAuth Secondary)

```
Priority 1: ANTHROPIC_API_KEY (user enters key in AgentMux settings)
Priority 2: CLAUDE_CONFIG_DIR isolation (user runs claude auth login via AgentMux)
Priority 3: apiKeyHelper script (for enterprise/rotating credentials)
```

This mirrors how the CLI itself handles precedence, and is fully sanctioned by Anthropic.

### 6.2 Storage Architecture

```
~/.agentmux/
├── auth/
│   ├── api-key.enc          # Encrypted API key (via OS credential store)
│   ├── claude-config/       # CLAUDE_CONFIG_DIR for OAuth path
│   │   ├── .credentials.json
│   │   └── settings.json
│   └── auth-state.json      # Which auth method is active
```

### 6.3 AgentMux Settings for the Claude Config Dir

Write this to `~/.agentmux/auth/claude-config/settings.json`:

```json
{
  "env": {
    "DISABLE_AUTOUPDATER": "1",
    "DISABLE_TELEMETRY": "1"
  }
}
```

This keeps the isolated directory clean and prevents auto-update telemetry from the embedded CLI.

## 7. Implementation Plan

### Phase 1: API Key Auth (Simplest, Ship First)

**UI:** In AgentMux auth settings pane, show:
- Text field: "Anthropic API Key"
- Link: "Get your API key at platform.claude.com/settings/keys"
- Validation: call `claude -p "hi" --max-turns 1` with the key and check exit code

**Storage:** Use Windows Credential Manager (`wincredential`) or macOS Keychain to store the API key encrypted at rest.

**CLI Spawn:**
```rust
fn spawn_claude_with_api_key(api_key: &str, prompt: &str) -> Command {
    let config_dir = agentmux_claude_config_dir(); // ~/.agentmux/auth/claude-config
    let mut cmd = Command::new("claude");
    cmd.env("ANTHROPIC_API_KEY", api_key)
       .env("CLAUDE_CONFIG_DIR", &config_dir)
       .env("DISABLE_AUTOUPDATER", "1")
       .args(["-p", "--verbose", "--output-format", "stream-json",
              "--include-partial-messages"]);
    cmd
}
```

### Phase 2: OAuth Auth via CLAUDE_CONFIG_DIR (Claude Pro/Max Users)

**Auth Flow:**
1. AgentMux calls: `CLAUDE_CONFIG_DIR=~/.agentmux/auth/claude-config/ claude auth login`
2. The `claude` CLI opens the user's browser to `https://claude.ai/oauth/authorize?...`
3. User approves in browser; CLI receives the callback; writes `.credentials.json`
4. AgentMux detects completion by polling `claude auth status --json`
5. AgentMux stores a marker that OAuth is configured

**UI:**
- "Connect your Claude account" button
- Opens an overlay ("Authenticating with Claude.ai...") while the auth is in progress
- On success: shows logged-in email and subscription type

**Checking auth:** AgentMux should poll `claude auth status --json` at startup and before spawning a new session.

**Token refresh:** Handled automatically by the `claude` CLI during its normal operation — AgentMux does not need to manage token refresh.

### Phase 3: Enterprise / Cloud Provider Auth

Support `CLAUDE_CODE_USE_BEDROCK`, `CLAUDE_CODE_USE_VERTEX`, `CLAUDE_CODE_USE_FOUNDRY` via Advanced Settings. Power-user feature.

### 8.2 OAuth Flow (Option A)

1. User clicks "Connect with Claude.ai →"
2. AgentMux spawns: `CLAUDE_CONFIG_DIR=<isolated> claude auth login` in a hidden terminal pane
3. AgentMux displays a waiting overlay: "A browser window has opened. Log in to Claude.ai to continue."
4. Browser opens `https://claude.ai/oauth/authorize?...`
5. User approves; browser redirects; CLI completes
6. AgentMux polls `claude auth status --json` every 500ms
7. On `loggedIn: true`, overlay clears; auth state saved; user is ready

1. User pastes API key
2. AgentMux validates: `ANTHROPIC_API_KEY=<key> claude -p "Hello" --max-turns 1`
3. If exit code 0: key stored in OS credential manager; auth state saved
4. If error: show error message ("Invalid API key" or "Quota exceeded")

### 8.4 Auth Status in the UI

AgentMux should show auth status in the status bar (or agent settings panel):
- "Claude: Connected (Pro)" — OAuth auth active with subscription type
- "Claude: API Key (Console)" — API key auth active
- "Claude: Not connected" — needs setup; clicking opens auth modal

## 9. How Credentials Are Used When Spawning `claude -p`

### 9.1 API Key Path

```rust
cmd.env("ANTHROPIC_API_KEY", stored_api_key)
   .env("CLAUDE_CONFIG_DIR", agentmux_config_dir)
```

The `CLAUDE_CONFIG_DIR` is still set to keep settings (like `DISABLE_AUTOUPDATER`) isolated, even though auth comes from the env var.

### 9.2 OAuth Path

```rust
cmd.env("CLAUDE_CONFIG_DIR", agentmux_config_dir)
// ANTHROPIC_API_KEY is NOT set, so claude falls through to OAuth credentials
// in $CLAUDE_CONFIG_DIR/.credentials.json
```

The claude binary reads `$CLAUDE_CONFIG_DIR/.credentials.json`, finds the `claudeAiOauth` block, and uses those tokens. Refresh happens automatically inside the claude process.

### 9.3 Guaranteed Isolation

The user's personal `~/.claude/` directory is **never consulted** when `CLAUDE_CONFIG_DIR` is set. Both auth paths set this env var, so there is zero cross-contamination between AgentMux sessions and the user's standalone `claude` CLI usage.

## 10. Trade-offs Analysis

### Option A: `CLAUDE_CONFIG_DIR` + `claude auth login` (OAuth)

| Aspect | Notes |
|--------|-------|
| User billing | Billed to their Claude.ai subscription (Pro/Max/Team) |
| UX friction | Medium: requires browser auth flow once |
| User familiarity | High: same credentials as claude.ai web |
| API surface | Uses claude.ai OAuth (subscription-gated) |
| Isolation | Complete: separate `.credentials.json`, separate keychain entry |
| Refresh | Handled by claude CLI automatically |
| Security | Good on macOS (keychain). Windows/Linux: plaintext JSON (mode 0600) |
| Third-party branding | None: user sees "Claude Code" in the OAuth consent screen |
| Approvals needed | None: uses existing Claude Code client ID |

### Option B: `ANTHROPIC_API_KEY` (Console API Key)

| Aspect | Notes |
|--------|-------|
| User billing | Billed per-token via Anthropic Console (pay-per-use) |
| UX friction | Low: paste key once |
| User familiarity | Medium: requires Console account, separate from claude.ai |
| API surface | Full API access, no subscription needed |
| Isolation | Complete: env var overrides all credentials |
| Refresh | No refresh needed (static key until rotated) |
| Security | AgentMux must store key encrypted in OS credential manager |
| Third-party branding | No OAuth UI; user just pastes a string |
| Approvals needed | None |

### Option C: Custom Anthropic OAuth App (Not Available)

| Aspect | Notes |
|--------|-------|
| Status | **Not available via self-service** |
| What it would give | Custom branding in OAuth consent screen ("Log in for AgentMux") |
| Path to get it | Contact Anthropic sales; likely only available to large enterprise partners |
| Alternatives | None that achieve the same UX |

### Option D: `apiKeyHelper` Script

| Aspect | Notes |
|--------|-------|
| Use case | Enterprise: rotating credentials from vault |
| Implementation | Write helper path to `$CLAUDE_CONFIG_DIR/settings.json` |
| AgentMux role | Must manage the helper script and its credential source |
| Complexity | High: AgentMux becomes a credential broker |
| Recommendation | Ship as advanced/enterprise feature later |

---

## 11. Summary Recommendation

**Ship in order:**

1. **API Key auth first** (Option B) — lowest friction, works for all users with any Claude access, zero Anthropic dependencies on our side. Covers the power-user / developer demographic immediately.

2. **OAuth via `CLAUDE_CONFIG_DIR`** (Option A) — covers Claude Pro/Max/Team subscribers who do not want to manage API keys and just want "log in with my Claude account". The OAuth consent screen will say "Claude Code" (since we use their shared client ID), which is acceptable and honest.

3. **Enterprise (Bedrock/Vertex/Foundry)** — advanced settings for teams already on cloud providers.

**Do not pursue:**
- Registering AgentMux as its own OAuth client — not self-service, Anthropic explicitly discourages this for third-party apps.
- Parsing/reusing the user's existing `~/.claude/.credentials.json` — would violate isolation and could corrupt the user's standalone Claude CLI session.

## 12. Key Environment Variables Reference

| Variable | Purpose for AgentMux |
|----------|---------------------|
| `CLAUDE_CONFIG_DIR` | **Always set** to `~/.agentmux/auth/claude-config/`. Isolates all state. |
| `ANTHROPIC_API_KEY` | Set when user has provided an API key. Takes precedence over OAuth. |
| `ANTHROPIC_AUTH_TOKEN` | Alternative to API key; used for bearer-token gateway setups. |
| `CLAUDE_CODE_OAUTH_CLIENT_ID` | Do not set; would require a registered Anthropic OAuth client. |
| `CLAUDE_CODE_OAUTH_TOKEN` | Advanced: inject a pre-fetched OAuth token directly (requires AgentMux to handle refresh). |
| `DISABLE_AUTOUPDATER` | Set to `1` in spawned processes to prevent claude CLI auto-updates during sessions. |
| `DISABLE_TELEMETRY` | Set to `1` to opt out of Statsig telemetry from embedded CLI. |

To verify `CLAUDE_CONFIG_DIR` isolation works correctly before implementing in Rust:

```bash

# Verify credentials written to isolated dir (not ~/.claude/)

CLAUDE_CONFIG_DIR=~/.agentmux/auth/claude-config \
```
